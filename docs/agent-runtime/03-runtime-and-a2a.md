# 03 — Runtime and Agent-to-Agent Communication

## Communication Model

A2A communication is not modeled as free-form chat. The core contract uses four
primitive message types:

| Type | Meaning | Initiator | Example |
|------|---------|-----------|---------|
| **command** | Intent to perform an action | any agent | `assign-task`, `start-research`, `request-review` |
| **event** | Fact that something happened | any agent or runtime | `task-accepted`, `artifact-produced`, `review-completed` |
| **status** | Canonical snapshot of task or work-item state | task store | `task:in-progress`, `us:code-review` |
| **artifact** | Result or reference to a result | producing agent | research report, PR reference, test result |

Commands express intent. Events express facts. Status is derived state. Artifacts are outputs.
These must not be conflated.

## Handoff Contract

Every handoff must include:

| Field | Required | Notes |
|-------|----------|-------|
| `owner` | Yes | Agent taking ownership |
| `requester` | Yes | Agent requesting the work |
| `goal` | Yes | What needs to be accomplished |
| `definition_of_done` | Yes | Explicit acceptance criteria |
| `input_artifacts` | If applicable | References to input context or prior work |
| `input_context` | If applicable | Inline context if no artifact |
| `callback_target` | Yes | Where the result should be returned |
| `priority` | Yes | Relative priority of this work |
| `deadline` | If applicable | SLA or expected completion time |

See the canonical schema in [reference/handoff-contract.md](reference/handoff-contract.md).

## Acceptance Semantics

A handoff is not complete at dispatch. It transitions through:

```
dispatched → pending → accepted | rejected → in-progress → done | failed | escalated
```

- **pending**: delivered, waiting for acknowledgment
- **accepted**: the receiving agent takes formal ownership; corresponding task moves to `Accepted`
- **rejected**: the receiving agent declines; reason must be provided; ownership reverts to requester

Until accepted, the requester retains accountability. `claimed` is not a v1 state —
use `accepted` for all ownership transfers.

## Callback Model

Every non-trivial task has an explicit callback target. A callback is not optional for
important work. Best-effort fire-and-forget without an explicit callback contract must
not be the default.

The callback is a first-class field in the handoff contract. When work completes,
the executor sends a result event to the callback target with:
- the task reference
- the outcome status
- a reference to output artifacts
- a brief summary or reason if the outcome is partial or failed

## Correlation and Causation IDs

All messages must carry:

| Field | Purpose |
|-------|---------|
| `correlation_id` | Groups all events belonging to one logical flow or request |
| `causation_id` | Points to the event that caused this event |
| `dedup_key` | Enables idempotent processing; safe to re-deliver |
| `version` | Contract version for the message type |

These are non-optional. They are required for replay, debugging, and deduplication.

### ID Generation Rules

| Field | Generated by | Format | Notes |
|-------|-------------|--------|-------|
| `correlation_id` | James (at start of each user request) or MC (at start of a scheduled flow) | UUID4 | All events in the same flow share this ID |
| `causation_id` | Sender | `event_id` of the triggering event | Sub-tasks from one handoff all have the same `causation_id` |
| `dedup_key` | Sender | `{event_type}:{task_ref}:{actor}:{attempt_number}` | Same on retry; ensures idempotency |
| `version` | Hardcoded | `"v1"` in v1 | No migration logic in v1 |

**Propagation rule:** When James creates a user request, he generates a `correlation_id`.
Every event triggered by that request — including subtasks Naomi creates, Amos' review
events, and callbacks — carries the same `correlation_id`. A new `correlation_id` is
only generated at the start of a new top-level request or scheduled flow.

## Idempotency

All message handlers must be idempotent with respect to `dedup_key`. Receiving the
same message twice must produce the same outcome as receiving it once.

This applies to:
- task acceptance
- status transitions
- artifact registration
- memory writes triggered by events
- callback delivery

## Detached vs Session-Bound Execution

The system supports two execution modes:

**Session-bound (synchronous short-turn)**
- short interactions needing quick acknowledgment or a short answer
- result returned within the same session context
- suitable for clarifications, quick lookups, simple delegation

**Detached (async durable)**
- background tasks with explicit callback
- retry and recovery semantics apply
- full event trail required
- suitable for implementation, research, review, verify

Detached runs are first-class. For important work, detached with a callback contract
is the correct default, not session-bound fire-and-forget.

## Example Flows

### James → Naomi (implementation handoff)

```
James sends:
  type: command / assign-task
  correlation_id: flow-123
  causation_id: user-request-456
  owner: naomi
  requester: james
  goal: "Implement feature X as described in US-42"
  definition_of_done: "All tasks under US-42 pass review and verify"
  input_artifacts: [us-42-ref]
  callback_target: james
  priority: normal
  execution_mode: detached
```

### Naomi → James (callback after completion)

```
Naomi sends:
  type: event / task-completed
  correlation_id: flow-123
  causation_id: naomi-task-789
  task_ref: us-42
  outcome: done
  artifacts: [pr-ref-101]
  summary: "Implementation complete, PR open for review"
  callback_target: james
```

### Naomi → Amos (review handoff)

```
Naomi sends:
  type: command / request-review
  correlation_id: flow-123
  causation_id: naomi-task-789
  owner: amos
  requester: naomi
  goal: "Code review for US-42 implementation"
  definition_of_done: "Review approved or comments returned"
  input_artifacts: [pr-ref-101]
  callback_target: naomi          ← operational callback: loop returns go to Naomi
  strategic_callback: james       ← strategic callback: Done and Escalation go to James
  priority: normal
```

Note the **dual callback pattern**: `callback_target` is Naomi for loop returns
(comments → In Progress → reassign Naomi). James is the strategic callback, notified
only on terminal outcomes (Done, Escalated). MC is responsible for routing these
two callbacks correctly — Naomi is not in the critical path for the James notification.

## Protocol Stance: Google A2A and MCP

The yaga.ai A2A protocol (commands / events / statuses / artifacts) is an internal
protocol, intentionally simpler than Google A2A (now in Agentic AI Foundation / Linux
Foundation, as of late 2025).

**Current stance:**
- We do not implement Agent Cards, JSON-RPC, or HTTP/SSE as per Google A2A spec
- We do not expose MCP endpoints in v1
- Our A2A semantics (explicit ownership, callback contracts, causation IDs) are
  compatible in spirit with Google A2A but are not wire-compatible

**When this changes:** If yaga.ai needs to interoperate with external agents or tools
that speak Google A2A or MCP, we will define a translation layer at the boundary.
The internal protocol will not be changed to match external specs.
